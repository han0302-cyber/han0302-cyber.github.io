13 Skills｜把你的 SOP 變成 AI 的技能

發表於 2026-07-07 更新於 2026-07-09 分類於 AI x Obsidian 系列 閱讀次數：

作者｜有澤法律事務所 林廷翰律師

💡 本篇重點
Skill 就是「一個資料夾＋一份 SKILL.md 說明書」，把你反覆使用的工作 SOP 寫成 AI 看得懂的作業程序——寫一次，之後一句話觸發，而且全所可以共用、品質高度穩定。

一、從「好用的提示詞」到「制度化的技能」

前幾篇你已經累積了不少好用的提示詞：Inbox 編纂、三色摘要、契約審閱……但提示詞有三個先天限制：

  1. 要一直複製貼上，或存在某個地方每次去翻；
  2. 品質會飄移——同一件事，這次講得完整、下次漏講一句，產出品質就浮動；
  3. 難以共享——你磨了三個月的提示詞，同事拿去用還是要學你的用法。

💡 小提示
先介紹一個中間解法：Claude Code 支援自訂斜線指令——把常用提示詞存成檔案（放在專案的 .claude/commands/ 資料夾），之後輸入 /指令名 就能叫出整段提示詞，90_SYSTEM/Prompts/ 裡的常用提示詞都值得做成指令。它解決了第一個問題（複製貼上），但「品質飄移」與「自動觸發」還是要靠本篇的主角——你可以把斜線指令理解為 Skill 的簡化版。

Claude Code 的 Skills（技能） 機制完整解決這三個問題。一句話定義：

Skill = 一個資料夾，裡面放一份 SKILL.md，內容是教 AI 完成某項特定任務的標準作業程序。

裝好之後，AI 會在對話中自動判斷何時該用它——你說「幫我整理這份判決」，它就自己套用裁判書摘要技能，不需要你提技能的名字。

對律師來說，最貼切的理解是：Skill 就是你寫給新進助理的 SOP 手冊。 差別在於，這位助理會穩定地按手冊執行，而且手冊可以複製給無限多位助理。本篇後段也會談到，這套機制如何回應師徒制「帶得慢、留不住」的傳承難題。

二、Skill 的解剖：以「裁判書摘要」為例

以下解剖筆者實際使用中的 judgment-summary（裁判書摘要）技能——它正是 實戰｜打造判決知識庫：從裁判書到 WIKI 索引 三色摘要的引擎。看懂它，就懂 Skill 的全部結構。

位置與組成

SKILL.md 的三個組成部分

第一部分：frontmatter（讓 AI 知道何時觸發）

description 是靈魂：AI 靠它判斷「現在這個任務是不是該用這個技能」。所以要把各種可能的說法都寫進去——包括使用者「沒有明講」的情況。

第二部分：工作流程（步驟化的 SOP）

judgment-summary 的流程段落規定：完整讀完裁判書（不能只看主文）→ 檢查是否已整理過（避免重複插入）→ 撰寫三個 callout 區塊 → 插入在 H1 標題之後 → 用編輯方式插入、不重寫全文。

注意這些步驟的顆粒度——跟你教新人的顆粒度一樣細。「法院見解多半藏在理由欄，不能只看主文」這種只有律師知道的細節，正是 Skill 的價值所在。

第三部分：格式模板與紅線原則

模板規定三色 callout 的精確格式（哪個用綠色 [!tip]、關鍵事實加螢光標記……）；原則段落則是紅線：不杜撰（摘要必須能在全文找到依據）、數字照抄（金額、日期、字號不可四捨五入）、不更動全文、沿用當事人稱謂、台灣法律用語。

換個角度看，這份 SKILL.md 的本質，是把一位資深律師「怎麼讀判決、怎麼做摘要」的內隱知識，外顯化成文件。寫 Skill 的過程本身就是知識管理。

三、動手做：兩條路徑做出你的第一個 Skill

做 Skill 有兩條路徑，依你手上有沒有「已經跑順的流程」來選。

路徑 A：先把流程跑順，再封裝成 Skill（推薦）

最自然的做法：某個工作流你已經用提示詞帶著 AI 做過幾次、格式與步驟都修到滿意——例如上一篇的契約審閱。此時在剛做完的那個對話裡接一句：

AI 會把「剛剛實際發生的流程」萃取成 SOP。這條路徑的好處是：Skill 從實戰長出來，每個步驟都經過你驗收，不是憑空設計；缺點是要等流程先成熟。日常建議以這條為主。

路徑 B：從零開始，用 skill-creator 引導生成

流程還沒實際跑過、但 SOP 已經在你腦中（或所內已有書面 checklist 可以直接搬上來）時，用官方的 skill-creator 引導式生成。它以外掛形式提供，先安裝一次（/plugin install skill-creator），之後輸入：

兩條路徑共同的收尾：審閱、試用、迭代

無論哪條路徑，AI 都會生成 SKILL.md 草稿給你確認。審閱看兩個重點：description 是否涵蓋你會講的各種說法；SOP 步驟是否符合你的實際習慣。然後拿一份契約實測（若在消費者方案環境，先「去識別化」；分級判斷見 資安與倫理｜律師使用 AI Agent 的紅線），產出不合意的地方直接說「以後審閱備忘錄的風險表格要加一欄『對方可能的抗辯』，請更新技能」——Skill 跟 CLAUDE.md 一樣是養出來的。

四、兩種視角的 Skill 清單（點子庫）

訴訟：

  • 裁判書摘要（本篇範例）
  • 判決檢索入庫（一句「幫我找工期展延的新判決」→ 透過法規資料庫 MCP 檢索、列候選清單經我確認 → 下載全文、套三色摘要、更新索引與爭點 WIKI——把 實戰｜打造判決知識庫：從裁判書到 WIKI 索引 做法四到第三步的整條產線一句話觸發；所需工具見 MCP｜把 AI 接上法規資料庫）
  • 開庭筆記整理（口語紀錄 → 正式筆記＋更新案件總覽）
  • 開庭準備包（實戰｜訴訟與非訟的日常工作流 工作流一的制度化——2026-015 案每次開庭前的那段準備包提示詞，寫成 Skill 後一句「準備明天的庭」即可觸發）
  • 卷證索引（新收卷宗 PDF → 命名歸檔＋建立索引筆記）
  • 書狀格式檢查（引用格式、當事人稱謂一致性、誤字）

非訟／企業法務：

  • 契約審閱備忘錄（本篇範例）
  • 意見書架構草稿（含依據強度標註）
  • 函釋摘要（主管機關函釋 → 重點／適用情境／機關立場三區塊）
  • 教育訓練教材產生器（主題 → 大綱＋講義＋測驗）
  • 中日／中英契約對譯檢查（術語一致性表）

原則：一個月內用到三次以上的流程，就值得寫成 Skill。 寫成 Skill 之後，第一百次執行仍遵循與第一次相同的完整流程（AI 仍可能偶有落差，但遠比口頭提示穩定）。

五、多人事務所的視角：從師徒制到制度化傳承

法律專業的傳承，長期依賴師徒制：資深律師帶著年輕律師改書狀、看契約，經驗一對一慢慢滲透。這套方式有效，但有兩個天生限制——帶得慢（一次只能帶一兩個人）、留不住（資深者退休或離開，經驗跟著走）。知識庫＋Skills 針對這兩個限制，提供了一套可操作的解法：

  • 經驗外顯化：資深律師「怎麼讀判決、怎麼審約、書狀的雷區在哪」這些內隱知識，寫成 SKILL.md 與 checklist 筆記——寫一次，全所每一位同仁的每一次任務，都等於在被他帶。
  • 回饋回流：年輕同仁使用中發現的新問題（新型條款、新的實務見解），經資深律師確認後更新回 Skill 與知識庫——師徒制從單向變成雙向、持續演化的制度。
  • 新人訓練改觀：新人報到第一天，讀 CLAUDE.md 知道所規、用 Skills 繼承方法、查爭點 WIKI 看前人研究——上手期從「半年摸索」縮短為「一週導覽」。
  • 人員異動的防波堤：律師離職、輪調、退休，帶走的是他自己；他寫進 Skill 與知識庫的方法，留在所裡繼續帶下一代。

實務作法：把技能資料夾放進共用 Vault，整個團隊的判決摘要、審約備忘錄從此同一個格式。共用 Vault 可放在所內網路磁碟，或以 Git（工程界通用的版本管理工具，能完整記錄「誰、何時、改了哪一行」，導入需要一點技術支援）同步——多人編輯的版本管理正是它的看家本領；Skills 與範本資料夾隨 Vault 一起發佈，就是全所的標準配備。

對資深律師而言，把 know-how 寫成 Skill 意味著角色的轉變：從「親手做每一件事的人」，變成「定義品質標準的人」。

對管理者而言：事務所真正的資產不是工時，是方法。 過去方法存在資深律師的腦中，難以盤點、難以複製；現在，它可以是一個看得見、傳得下去的資料夾。

💡 小提示
Codex 使用者：Codex 自 2025 年 12 月起也支援同構的 Skills 機制——同樣是資料夾裡放一份 SKILL.md（含 name 與 description），個人技能放 ~/.codex/skills/、專案技能放 .codex/skills/，依 description 自動觸發。本篇的方法與範例可以原樣搬過去；AGENTS.md（等同 CLAUDE.md 的常駐規則）與 Skills（隨任務觸發的 SOP）兩者搭配使用。

六、他山之石：Anthropic 官方的法律 Skills——Claude for Legal

Anthropic 自己也把法律工作流寫成了 Skills。2026 年 2 月先推出法律插件（legal plugin）；同年 5 月擴大為 Claude for Legal——一整套開源（Apache 2.0，公開於 GitHub）的法律插件套件，內容值得台灣律師認識：

它包含什麼

十二個執業領域插件——商務契約（vendor 契約、NDA 分流）、公司法務（併購盡調、交割清單）、勞動、隱私（DPA 審閱、DSAR 回覆）、產品、法遵監管、AI 治理、智財（商標檢索、FTO、DMCA）、訴訟（案件收案、爭點對照表、書證特權清單），甚至還有法律扶助診所與法學生學習兩個公益取向的插件——合計上百個技能，並提供二十多個法律科技系統的 MCP 連接器（DocuSign、Ironclad、Everlaw、Relativity、Thomson Reuters Westlaw 等）。

它怎麼運作：跟本篇教的完全同構

拆開它的原始碼，每個技能就是一份 SKILL.md（frontmatter 寫觸發條件、內文寫 SOP），與本篇第二節解剖的結構一模一樣。最值得注意的是它的啟動訪談（cold-start interview） 設計：安裝後第一件事是接受 AI 訪談——你的執業領域、審核放行標準、風險容忍度、慣用格式——訪談結果寫成一份「執業檔案」（本質就是 CLAUDE.md），之後所有技能都讀取這份檔案執行。這正是本系列「CLAUDE.md 定規矩、Skills 定方法」架構的官方版。它的紅線原則也與本系列一致：每個引用都附出處、未經查證的來源標記待核、所有產出定位為「供律師審閱的草稿」，法律判斷的責任在律師。

與台灣法律環境的差異

直接拿來用之前，要看清楚三個落差：

  1. 法域：整套內容以美國法與美式執業為底——州法差異的提示、privilege log、FTO、DMCA 這些概念與工作流，是普通法系與美國法制的產物；playbook 的預設立場（例如 NDA 的市場行情條款）也是美國市場慣行。臺灣的成文法體系、函釋實務、書狀格式，它一概不知道。
  2. 資料庫：它的查證連接器接的是 Westlaw、CourtListener 等美國法源；全國法規資料庫與司法院裁判書系統不在其中——這一塊正是本系列 MCP｜把 AI 接上法規資料庫 自建的 taiwan-legal-db 在補的洞。
  3. 語言與用語：技能的模板、術語、產出格式都是英文與美國法律用語，中文化不只是翻譯，而是整套 SOP 的在地重寫。

給台灣律師的啟示

  1. 方向已經確認：全球最前沿的 AI 公司把「AI 進入法律業」的解答押在 Skills＋MCP＋執業檔案這套架構上——本篇的 judgment-summary 與契約審閱技能，與官方路線同構，這條路值得繼續投資。
  2. 架構可以借鑑、內容必須自建：它是開源的，啟動訪談、風險分級、紅線原則這些設計都可以拆開來學；但台灣法的技能，Anthropic 不會幫我們寫，台灣版的 Claude for Legal 只能由懂台灣法的律師自己動手。目前沒有台灣在地的現成套件，所內的 Skill 需要自力撰寫，寫出來的也就完全貼合自身的業務。
  3. 免責立場的印證：工具再完整，產出永遠是草稿，判斷與責任永遠在律師——與本系列反覆強調的原則一致。

小結

  • Skill = 一個資料夾＋一份 SKILL.md，把反覆使用的 SOP 寫成 AI 能自動觸發的作業程序。
  • 兩條製作路徑：從跑順的流程封裝（日常首選），或用 skill-creator 從零生成；完成後都要實測與迭代。
  • 一個月內用到三次以上的流程，就值得寫成 Skill；多人事務所可藉此把師徒制的經驗外顯化、留在所內。
  • 下一篇處理「引用的法條判決到底對不對」：讓 AI 直接連上法規資料庫查證。
