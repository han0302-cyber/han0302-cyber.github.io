作者｜有澤法律事務所 林廷翰律師

> **💡 本篇重點**
> CLAUDE.md 是 AI 每次啟動都自動讀取的「到職須知」。標準做法是三層分工——全域寫「你是誰、通用規矩」，知識庫（Vault 根目錄）寫「這個知識庫的規則」，個別案件寫「本案的特殊要求」；案件背景另立 background.md 用 `@` 語法引入。寫一次，之後每次啟動都自動適用。


<!-- more -->
## 一、問題：每次都要重新交代一遍

用過 ChatGPT 的人都有這個經驗：每開一個新對話，都要重講一次「請用繁體中文」「我是律師」「引用法條要附條號」。因為 AI 沒有記憶，每次對話都像換了一位新助理。

如果事務所每天來一位新助理，沒有人會每天早上重新交代一次書狀格式、引用慣例、保密規矩；通常的做法是寫一份到職須知，新人來了自己看。

CLAUDE.md 就是這份到職須知。Claude Code 啟動時會**自動讀取**它並在整個工作過程中遵守，不需要任何設定。（Codex 的對應機制是 `AGENTS.md`，觀念完全相同；想讓兩套工具共用，可以只維護一份 AGENTS.md，CLAUDE.md 裡寫一行「請遵守 AGENTS.md」。）

## 二、標準架構：三層分工

CLAUDE.md 可以放在不同位置，效力範圍不同，**疊加適用**——跟法規的位階與特別法優先，邏輯相通。載入時機有一個細節：全域與知識庫層在啟動時就讀入；子資料夾層則是**用到才載入**——AI 實際讀取該資料夾內的檔案時，那一層的 CLAUDE.md 才會生效（所以案件層的規則，放進案件資料夾即可，不必擔心拖慢其他工作）：

| 層級 | 位置 | 該寫什麼 |
| --- | --- | --- |
| **全域（你這個人）** | `~/.claude/CLAUDE.md`（家目錄） | 你是誰、語言、文書偏好、通用紅線——**無論在哪個資料夾工作都適用的事** |
| **知識庫（整個 Vault）** | Vault 根目錄的 `CLAUDE.md` | 知識庫結構、筆記格式、編纂規則——**整個知識庫通用的事** |
| **子資料夾（個別案件）** | `10_MATTERS/某當事人/某案件/CLAUDE.md` | 該案件的特殊要求 |
| 補充：個人（僅自己生效） | Vault 根目錄的 `CLAUDE.local.md` | 團隊共用 Vault 時你個人的偏好，同樣自動讀取 |

分層的判準一句話：**「換一個 Vault 還成立的，往上放；只跟這個案子有關的，往下放。」**

### 全域層實例：一位涉日律師的 `~/.claude/CLAUDE.md`

以下以筆者實際使用的全域設定為例（節錄），這一層寫的是「無論在哪個資料夾工作都不變」的事：

```markdown
# 關於我
- 角色：台灣執業律師，主要處理涉日跨境法律業務
- 專業領域：再生能源法令、個人資料保護法、AI

# 語言
- 預設以繁體中文回覆。
- 撰寫日文法律文書時直接使用日文，不需額外翻譯成中文。

# 回覆風格
- 簡單問題簡答，不需寒暄或鋪陳。
- 法律分析涉及具體法條、判例字號或數據時，
  若無法確認請明說，不要臆測或杜撰。

# 法律文書偏好
- 日文文書使用「ですます調」，用語精確、句構清晰。
- 引用法條時註明法規全名與條號。
- 日文文書一律使用西元紀年；判決字號中的年號（令和、平成、
  民國）也一律換算為西元表示。

# 工具
- 處理 docx 追蹤修訂時，檢閱者名稱設為「ST」。
- 輸出任何台灣法條條文或判決要旨前，必須先用法規資料庫
  MCP 工具核對；查無資料時明說「未能於資料庫確認」，
  不得憑記憶引用。
```

留意幾個寫法：每一條都是**可執行的行為規則**，不是抽象期許；「不得臆測、查無資料要明說」這種職業倫理層級的紅線，直接寫成 AI 的行為限制；連「追蹤修訂檢閱者名稱」這種瑣碎但每次都要交代的細節，寫進去就一勞永逸。

### 知識庫層實例：Vault 根目錄的 `CLAUDE.md`

```markdown
# 關於這個知識庫
台灣執業律師的法律知識管理 Vault（Obsidian）。

# 結構（雙軌，詳見各資料夾）
- 00_INBOX：新材料暫存，等待編纂
- 10_MATTERS：進行中案件，第一層以當事人分類（案件軌）
- 20_AREAS：法領域爭點筆記＋MOC（研究軌）
- 30_RESOURCES：判決、法規筆記、文獻
- 40_ARCHIVE：已結案封存
- 90_SYSTEM：範本與常用提示詞

# 筆記規則
- 新筆記一律套用 90_SYSTEM/Templates 的對應範本。
- 判決筆記檔名 = 完整判決字號。
- 提到 Vault 中已有筆記的判決、法條、爭點，必須加 [[wikilink]]。
- 整理新材料後，更新對應領域的 MOC 與判決索引。

# 紅線
- 不改動 40_ARCHIVE 內任何檔案；刪除任何檔案前先徵求同意。
- AI 產出的法條與判決引用一律標註「（待核對）」。
- 對外文書草稿開頭標明「AI 草稿，未經律師審閱」。
```

### 案件層實例：個別案件資料夾的 `CLAUDE.md`

```markdown
# 本案工作規則
- 本案所有書狀草稿存入 02_我方書狀/，檔名附版本號。
- 本案當事人資訊僅供案內使用，產出對外文件時一律以代稱表示。
- 案件狀態（庭期、待辦、時間軸）有更新時，寫入 00_案件總覽.md。
```

寫的都是「只在這個案子成立」的要求：檔名慣例、代稱規則、狀態更新的去處。這種檔案每案大同小異，建議做成**範本**（放 `90_SYSTEM/Templates/`），開新案建資料夾時帶入再依個案微調，不必每次從零寫起。

## 三、進階模式：案件背景檔（background.md）

案件層 CLAUDE.md 解決了「規則」，但 AI 要在這個案子裡工作，還缺一塊：**理解案件脈絡**——當事人與代理立場、案情梗概、核心爭點。這些內容較長，全部塞進 CLAUDE.md 會讓到職須知變得臃腫。

**建議做法是「引用式」架構**：CLAUDE.md 保持精簡、只寫規則，背景資料獨立成檔，再用 Claude Code 官方支援的 **`@` 引入語法**掛進來——CLAUDE.md 裡寫 `@檔名`，AI 讀取 CLAUDE.md 時就會把該檔案內容一併載入。（說明：`@` 引入是官方機制；「background.md」這個檔名則是本系列的自訂慣例，你可以取任何名字，重點是每個案件用同一個。）

```
10_MATTERS/甲公司/2026-015_工程款_甲v乙/
├── CLAUDE.md          ← 只寫規則，兩三行
├── background.md      ← 案件背景（當事人、案情梗概、爭點框架）
└── ……
```

於是上一節的案件 CLAUDE.md 變成這樣——開頭新增「案件背景」一節放引入指令（`@` 引入放在檔案任何位置都有效，另立標題是讓人和 AI 都一眼看出這段載入的是什麼），原本的狀態更新規則也升級成「分流規則」：

```markdown
# 案件背景
@background.md

# 本案工作規則
- 本案所有書狀草稿存入 02_我方書狀/，檔名附版本號。
- 本案當事人資訊僅供案內使用，產出對外文件時一律以代稱表示。
- 案件狀態（庭期、待辦、時間軸）更新寫入 00_案件總覽.md；
  案情或策略的實質變動才更新 background.md。
```

這個設計的好處：

1. **關注點分離**：規則歸 CLAUDE.md、背景歸 background.md，各自精簡；
2. **背景檔可以請 AI 代管**：案情有實質變動時（追加聲明、和解方向改變），說一句「請更新 background.md」；
3. **一份背景多處使用**：background.md 同時也是你自己的案件摘要筆記，人和 AI 讀同一份。

> **💡 小提示**
> 另一種寫法：在**全域** CLAUDE.md 裡寫一條通則——「進入任何案件資料夾工作時，若該資料夾內有 background.md，請先讀取它再開始」，就不必在每個案件資料夾重複寫 `@background.md`。兩者比較：
> - **`@` 引入**：官方機制，**保證載入**；要在每個案件資料夾寫一行。
> - **全域通則**：寫一次、處處適用；但依賴 AI 記得去做，非保證。
> - 建議：固定結構的案件資料夾用 `@` 引入為主，全域通則作為補充保險，兩者並用最穩。

規則中反覆出現的 `00_案件總覽.md`，是案件資料夾的另一份核心檔案——案件的「儀表板」，記庭期、待辦、時間軸這些每週在變的狀態。它與 background.md 怎麼分工，留到下一篇 [檔案整理方法論｜案件與研究領域的雙軌知識系統](/2026/07/07/file-organization-methodology/) 介紹它時再細說；這裡先記一句就夠：**半年不會變的寫 background.md，每週在變的寫 00_案件總覽.md。**

## 四、撰寫與維護心法

**1. 讓 AI 幫你寫第一版。** Claude Code 內建 `/init` 指令，會分析資料夾內容自動生成初版 CLAUDE.md；或直接說「請觀察這個 Vault 的結構，起草一份 CLAUDE.md，草稿給我看過再存」。

**2. 用「#」隨手補充。** 在 Claude Code 對話中，以 `#` 開頭輸入一句話（例如 `# 以後判決筆記都要加裁判日期屬性`），它會問你要存進哪一層的 CLAUDE.md——這是「養」到職須知最順手的方式。也可用 `/memory` 指令直接開啟編輯。

**3. 從短開始，錯了就補。** 第一版十行就夠。每次 AI 做了你不想要的事，回頭加一條規則。三個月後，這份檔案就是你的 AI 管理經驗結晶。

**4. 定期「屋況檢查」。** 規則多了會互相打架或過時，每季問一次：「請檢查我的 CLAUDE.md 有沒有互相矛盾、過時或太模糊的規則，提出精簡建議。」

## 五、實際效果對比

**沒有 CLAUDE.md**：「幫我整理這則判決」→ 簡體用語混入、自創格式、沒有連結、引用錯了也不標註。

**有三層 CLAUDE.md**：同一句話 → 套用判決範本、台灣法律用語、自動連結相關爭點、更新 MOC 與索引、引用標「待核對」、還記得你的檢閱者名稱要設「ST」。

**差別不在 AI 變聰明了，而在你把要求講清楚了——而且只需要講一次。**

## 小結

- 三層架構：全域寫「你是誰」、知識庫寫「這裡的規矩」、案件寫「本案特殊要求」。
- 背景資料獨立成 background.md，用 `@background.md` 引入；全域通則可當補充保險。
- 四個維護工具：`/init` 起草、`#` 隨手補充、`/memory` 編輯、定期屋況檢查。
- 紅線（查證、保密、草稿標示）務必入文——這是職業紀律的程式化。
