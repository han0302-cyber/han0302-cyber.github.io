14 MCP｜把 AI 接上法規資料庫

發表於 2026-07-07 更新於 2026-07-11 分類於 AI x Obsidian 系列 閱讀次數：

作者｜有澤法律事務所 林廷翰律師

💡 本篇重點
MCP 是讓 AI 用統一方式連接外部資料庫與工具的開放標準。接上「台灣法規資料庫 MCP」之後，AI 引用法條不再單憑記憶臆測，而是即時查證後回答——這是對付幻覺最有效的工程手段之一。

一、律師的痛點：AI 幻覺的執業風險

AI 的幻覺對一般人是瑕疵，對律師是執業風險：編造判決字號、把廢止條文當現行法、虛構函釋內容。原因在本質——語言模型是「文字接龍機器」，它要的是「接得順」，不是「接得對」。靠模型自己「記住」全部法規，永遠不可靠；正解是讓它「當場去查」。

問題是：AI 要怎麼「去查」？它需要一條連接外部資料庫的管道。這條管道的業界標準，就是 MCP。

二、MCP 是什麼：AI 的萬國轉接頭

MCP（Model Context Protocol，模型上下文協定） 是 Anthropic 於 2024 年 11 月發布的開放標準，2025 年 12 月捐贈給 Linux 基金會旗下新成立的 Agentic AI Foundation 中立治理；OpenAI、Google、Microsoft 均已採用，是名符其實的業界共通標準，不是單一廠商的私有技術。

它解決的問題一句話就能講完：外部服務（司法院系統、法規資料庫、事務所 DMS）各有各的連接規格，AI 不可能一一遷就；MCP 就是節名說的萬國轉接頭——AI 只要學會這一種規格，就能接上所有服務。一個「MCP 伺服器」就是一個裝好的轉接頭，AI 連上它，就獲得它提供的查詢工具。

對使用者而言，知道這些就夠用了；MCP 與工程師熟悉的 API 是什麼關係、差異何在，不影響操作，留待文末附論。

三、實例：台灣法規資料庫 MCP

以筆者日常使用的「台灣法規資料庫 MCP」（taiwan-legal-db）為例，它讓 Claude Code 直接查詢：

  • 大法官解釋（釋字）與憲法法庭裁判（憲判字）

它還有第二個身分：檢索引擎。同一套工具不只被動核對你給的引用，也能主動出去撈——「請檢索工期展延相關的近三年判決，列清單供我挑選」，AI 就直接查裁判書系統、把候選判決帶回來（實戰用法見 實戰｜打造判決知識庫：從裁判書到 WIKI 索引 的做法四）。查證與檢索，是同一套工具的兩面。

驗證與日常管理

裝好後重新啟動 Claude Code，問一句「你現在有哪些工具可用？」——看到 taiwan-legal-db 的查詢工具（法規條文、裁判書、釋字與憲判字等八個），就是接好了。幾個實用指令：

  • claude mcp add：新增伺服器；claude mcp remove：移除
  • claude mcp list：查看目前接了哪些；對話中輸入 /mcp 可檢查連線狀態
  • 設定範圍分三層：local（只有你、只在這個專案）、project（隨資料夾共享給團隊）、user（你的所有專案）——與 CLAUDE.md 的三層邏輯一致。

其他 MCP 伺服器的安裝邏輯相同：依開發者提供的說明操作，或照路徑 A 把專案網址交給 AI（照例：看不懂就整段貼給 AI 問）。遠端服務（如下一節的 M365、Notion 連接器）則多以 HTTP 方式接入：claude mcp add --transport http <名稱> <網址>，接上時會引導你完成該服務的登入授權。

五、律師值得認識的其他 MCP

法規資料庫解決「查證」；下面三個連接器解決另一件大事——把 AI 接上你每天工作其中的信箱、行事曆與工作區。它們多由原廠或 Anthropic 官方提供，屬於「遠端 MCP」：不用在自己電腦安裝程式，接上時會跳出該服務的登入授權畫面（OAuth），你用自己的帳號授權、也隨時可以撤銷。

Microsoft 365：律師事務所的主場

事務所多半整個工作環境都在 M365 上。接上 Microsoft 365 連接器後，AI 可以檢索 Outlook 郵件與行事曆、搜尋 SharePoint／OneDrive 文件、查 Teams 對話。對律師的實際意義：

過去這是「開 Outlook 逐封翻、複製貼上」的苦工；接上之後，郵件往來直接成為知識庫的原料。開庭與會議期日也可以請 AI 對照行事曆檢查衝突。

Google Workspace：Gmail、日曆與雲端硬碟

用 Google 生態的個人執業者或中小所，對應的是 Gmail／Google 日曆／Google Drive 連接器，郵件整理、期日核對、文件讀取的用法與 M365 段所述同構。

Notion：案件管理看板的雙向同步

不少事務所用 Notion 管案件進度、開會紀錄。接上 Notion MCP 後，AI 能讀寫你的工作區：開完庭一句話，同時更新 Obsidian 的開庭筆記與 Notion 上的案件看板；週會前請 AI 把本週各案件的 Notion 進度彙整成一頁報告。知識庫（Obsidian）與協作平台（Notion）不再需要人工兩邊抄。

其他值得知道的類型

MCP 類型 功能 使用場景
網頁抓取類 讀取指定網頁內容 追蹤主管機關公告、公報
瀏覽器自動化類（如 Playwright） 讓 AI 實際操作瀏覽器：點擊、填表、下載 「到司法院裁判書系統查○○案由的近期判決並逐筆存檔」
檔案轉換類 PDF、Word 轉檔 卷證資料進 Vault 的前處理

⚠️ 接信箱與文件庫之前，先想資料流
信箱與文件庫是機密濃度最高的資料源：授權 AI 讀取，代表檢索到的內容會送到模型端處理。三個守則：(1) 依 資安與倫理｜律師使用 AI Agent 的紅線 的方案分級——用商用方案（Team／Enterprise）或 API 的環境再接，消費者方案不要接事務所信箱；(2) 授權範圍能窄就窄（例如只授權特定資料夾／標籤）；(3) 這類連接器天生是「唯讀優先」——讓 AI 讀郵件做整理沒問題，要它代發郵件、代改正式文件，務必保留人工確認那一關。

判斷要不要接一個 MCP 的思考順序：(1) 這個資料源是否經常查詢？(2) 接上後資料會流向哪裡？(3) 開發者可信嗎？——跟事務所評估委外廠商同一套邏輯。MCP 伺服器是第三方程式，只接必要的、可信的。

六、兩種視角的 MCP 用例

訴訟：書狀引用全面初核。以 2026-015 案的上訴理由狀為例——「請核對這份上訴理由狀草稿中所有法條引用與判決字號，逐一用法規資料庫與裁判書系統確認，產出核對報告：✅ 一致／⚠️ 條文已修正／❌ 查無此判決」。對造書狀也可以照樣核一遍，有時會有意外收穫。

非訟／企業法務：法規變動監控與意見書查證——意見書引用的每一條法規即時核對現行條文；客戶問題涉及的子法、函釋，先讓 AI 從資料庫撈全再開始分析。搭配 實戰｜訴訟與非訟的日常工作流 工作流五的「依據強度標註」，意見書的查證流程就完整了。

小結

  • MCP = AI 連接外部世界的統一標準；律師的第一個 MCP 應該是法規資料庫。
  • 安裝兩條路：把 GitHub 網址貼給 AI 請它代勞，或 pip install ＋ claude mcp add 兩行指令。
  • 黃金組合：MCP 工具 + CLAUDE.md「引用必先查證」規則。
  • 接 MCP 像選委外廠商：必要性、資料流向、可信度。
  • 知識管理主線到此完整。下一篇回答最後、也最重要的問題：這一切的邊界在哪裡。

附論：MCP 與 API 的差異——給想多懂一層的讀者

正文說 MCP 是「站在 API 前面的萬國轉接頭」，這裡把兩者的關係講完整。每個服務（司法院系統、法規資料庫、事務所 DMS）各自對外開的「門」就是 API（應用程式介面），規格五花八門，傳統上要由工程師寫程式去一一串接。MCP 不是要取代 API，而是站在各種 API 前面，把它們統一包裝成 AI 看得懂的樣子。 具體差異在四個層面：

傳統 API MCP
設計給誰用 工程師——要寫程式才能呼叫 AI 模型——在對話中自主呼叫
整合成本 每接一個服務，就要為它客製一次串接程式 統一協定，接上即用；AI 學會一種規格，就會用所有工具
使用說明 人類閱讀的技術文件 工具自帶機器可讀的說明——AI 連上的瞬間就知道「有哪些工具、各做什麼、要給什麼參數」
誰決定何時呼叫 開發者預先把流程寫死 AI 依當下任務自行判斷「何時查、查什麼、查幾次」

最後一列是關鍵差異：傳統 API 的世界裡，「查法規」這個動作必須被某個工程師預先寫進某套軟體；MCP 的世界裡，你只要說「幫我確認這條的現行條文」，AI 自己決定呼叫查詢工具——對不寫程式的律師，這是「根本用不到」與「裝好就能用」的差別。
